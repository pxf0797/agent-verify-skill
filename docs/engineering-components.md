## 一、方法论回顾：四块积木

视频作者提出了一套借鉴芯片验证的 AI Agent 工程化体系，包含四个相互配合的组件：

| 组件 | 一句话职责 | 芯片验证对应概念 |
|------|-----------|----------------|
| 1. 前后端分离 | 让 Agent 脱离人眼人手，独立运行并被 AI 调用 | 测试平台（Testbench）与 DUT 分离 |
| 2. 双层裁判 | 分层判定 Agent 行为质量：硬的归断言，软的归 LLM | Assertion + Scoreboard |
| 3. 回归测试 | 积累可重复执行的测试用例，每次改动自动复查 | Regression Test Suite |
| 4. Feature Flag | 新功能开关对比，量化每个改动的影响 | A/B 硅前验证（Pre-silicon A/B） |

## 二、可自动化操作提取与评估

### 评估维度说明

| 维度 | 标记 | 含义 |
|------|------|------|
| 自动化程度 | 自动 / 半自动 / 需人工 | 运行时是否需要人参与 |
| 实现复杂度 | 低 / 中 / 高 | 代码量 + 集成难度 + 基础设施依赖 |
| 技能要求 | 低 / 中 / 高 | 使用者需要什么背景知识 |
| 独立价值 | 核心 / 有用 / 锦上添花 | 单独拿出来用有没有意义 |

### 2.1 前后端分离组件（6 个操作）

> 目标：系统脱离人的眼睛和 UI 后仍能独立运行

#### 操作 1.1：Agent CLI 封装器

将 Agent 核心逻辑封装为一个命令行工具，接受结构化输入，产生结构化输出。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 一次封装，后续全部 CLI 调用 |
| 实现复杂度 | **中** | 需要解析现有 Agent 入口（Web/API），重写调度层；若已用 Vercel AI SDK 等框架，额外解耦成本 |
| 技能要求 | **中** | 需要理解现有 Agent 架构的入口→调度→输出链路 |
| 独立价值 | **核心** | 这是所有后续自动化的基础--没有 CLI，回归测试、LLM 裁判、闭环都无法挂接 |
| 可实现性评分 | **8/10** | 纯工程重构，无研究风险 |

**输入**：自然语言任务描述 + JSON 配置参数
**输出**：{workflow_steps: [...], tool_calls: [...], final_output: "...", execution_log: "..."}
**典型实现方式**：将 Agent 的 `run()` 函数包装为 CLI entrypoint（Python click/typer，Node.js commander）

---

#### 操作 1.2：结构化中间态日志输出

Agent 运行过程中，每一步的工具调用、中间结果、状态转换都以结构化格式（JSON/JSONL）输出，供后续分析。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 运行时自动产生，无需额外触发 |
| 实现复杂度 | **低** | 在工具调用/状态转换处插入 logger（OpenTelemetry span 或简单 JSON logger） |
| 技能要求 | **低** | 了解 Agent 的执行流程即可 |
| 独立价值 | **核心** | 没有结构化日志，双层裁判和回归测试都无数据可消费 |
| 可实现性评分 | **9/10** | 技术成熟，本质是结构化日志 |

**输出格式（建议）**：
```json
{
  "run_id": "uuid",
  "timestamp": "ISO8601",
  "step": 3,
  "step_type": "tool_call | llm_response | state_change | error",
  "tool_name": "search_database",
  "tool_input": {...},
  "tool_output": {...},
  "llm_response": "...",
  "state_snapshot": {...}
}
```

**关键约束（视频作者强调）**：日志必须"让另外一个 AI Agent 能看懂" —— 所以格式必须是结构化、自描述的，不能是纯文本 human-readable 日志。

---

#### 操作 1.3：ACI 接口规范定义

定义一套 Agent Computer Interface（ACI）规范，让 Codex 等 Coding Agent 可以程序化调用你的 Agent、读取运行信息。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **半自动** | 接口定义是人工设计的，但定义后调用是全自动的 |
| 实现复杂度 | **中** | 定义 OpenAPI/JSON Schema 接口规范；需要设计合理的资源模型 |
| 技能要求 | **中** | 需要 API 设计经验 + 对 Agent 能力的理解 |
| 独立价值 | **核心** | 如果没有规范化的接口，Coding Agent 无法"看懂"你的系统。视频引用普林斯顿 SWE-Agent 论文：接口设计 > 模型能力 |
| 可实现性评分 | **7/10** | 设计方案需要迭代，对小型 Agent 可能 over-engineering |

**规范应包含**：
- Agent 能力端点（有哪些可调用的功能）
- 输入参数 Schema（JSON Schema）
- 输出格式（workflow trace + final result）
- 中间状态查询端点（按 run_id 获取完整执行历史）
- 错误响应格式

#### 操作 1.4：Agent 黑盒健康检查

定时用一组固定输入探测 Agent 后端是否存活、响应延迟是否正常。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 定时 cron/CI 触发 |
| 实现复杂度 | **低** | 本质是 ping 端点 + smoke test |
| 技能要求 | **低** | 基本运维知识 |
| 独立价值 | **有用** | 先知道系统活着，再谈质量 |
| 可实现性评分 | **9/10** | 标准运维实践 |

---

#### 操作 1.5：前端瘦身为纯监控看板

将前端从"操作入口"降级为"只读监控 + 人工干预逃生舱"。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **需人工** | 前端运营仍需人看，但不再是人操作的必经之路 |
| 实现复杂度 | **中** | 需要重构前端数据流：从"用户输入→Agent"改为"Agent 后端推送→前端展示" |
| 技能要求 | **中** | 前端开发 + WebSocket/SSE 实时通信 |
| 独立价值 | **有用** | 对于有 UI 的系统，这步不小但能大幅降低后续测试摩擦 |
| 可实现性评分 | **6/10** | 重构已有 UI 的成本取决于原始架构质量 |

**注意**：作者明确说这步"花了不少力气"，尤其用了 Vercel AI SDK 的紧耦合场景。对纯 API Agent，这步可能不需要。

#### 操作 1.6：MCP 备选方案评估器

如果前后端分离成本太高，评估是否可用 MCP + 浏览器自动化（Playwright/Puppeteer）作为过渡方案。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **半自动** | MCP 脚本自动执行，但 corner case 需人工处理 |
| 实现复杂度 | **中** | 编写 MCP 操作脚本 |
| 技能要求 | **低** | 浏览器自动化基础 |
| 独立价值 | **锦上添花** | 作者明确放弃了这条路（authorization/corner case/效率），但对简单场景可能是够用的短期方案 |
| 可实现性评分 | **7/10** | 技术成熟但长期维护成本高 |

### 2.2 双层裁判组件（4 个操作）

> 目标：让人工"感觉对不对"变成自动化"精确判定 + 量化打分"

#### 操作 2.1：确定性断言引擎

可配置的硬性规则检查器：工具是否触发、中间状态记录数是否在预期范围、关键步骤是否被覆盖。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 每次测试运行后自动执行 |
| 实现复杂度 | **中** | 核心是规则引擎 + 断言 DSL；复杂点在于覆盖哪些检查项 |
| 技能要求 | **中** | 需要理解 Agent 的行为语义，才能编写有意义的断言规则 |
| 独立价值 | **核心** | 零成本、绝对可靠、即时反馈--这是整个验证体系的底线 |
| 可实现性评分 | **8/10** | 技术本身不复杂（规则匹配），但断言编写者需要领域知识 |

**断言类型建议**：

| 断言类型 | 示例 | 实现方式 |
|----------|------|----------|
| 工具调用断言 | "search_tool 必须被调用至少 1 次" | 扫描 workflow trace 中的 tool_call 事件 |
| 调用参数断言 | "search_tool 的 query 参数不能为空" | 检查对应 tool_call 的 input 字段 |
| 状态断言 | "步骤 3 之后，内部状态 items_count > 0" | 检查 state_snapshot |
| 路径断言 | "workflow 必须经过计划→执行→验证三个阶段" | 状态转移序列匹配 |
| 硬性规则断言 | "final_output 中必须包含 action_items 字段" | JSON Schema 验证 |
| 时间断言 | "单步 tool_call 耗时 < 30s" | 时间戳差值检查 |

---

#### 操作 2.2：LLM 裁判/评分器（Supervisor）

用一个与开发隔离的 LLM 对 Agent 的输出进行量化打分，处理断言引擎覆盖不了的模糊判断。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 每次测试运行后自动调用 |
| 实现复杂度 | **高** | 评分 prompt 设计、校准、一致性保证、成本控制都是实际工程难点 |
| 技能要求 | **高** | 需要理解 LLM 评估的局限性（偏好偏差、位置偏差、不一致性）并设计缓解措施 |
| 独立价值 | **核心** | 对 AI Agent 这种不确定性系统，没有 LLM 裁判就无法量化"模糊的好/坏" |
| 可实现性评分 | **5/10** | 技术上能做，但评委 LLM 本身的可靠性是开放问题 |

**关键设计约束（视频作者强调的必须遵守的原则）**：

1. **上下文隔离**：LLM 裁判不知道代码实现细节，"它的眼睛里面只有对正确的预期"
2. **量化打分而非二元判断**：不给 Pass/Fail，给 0-10 的连续分或 (-1, +1) 的变化趋势，回答"好了多少/差了多少"
3. **只知正确预期**：输入只有：原始任务 + Agent 输出 + 预期行为描述

**评分维度建议**：

| 评分维度 | 说明 | 示例问题 |
|----------|------|----------|
| 准确性 | 输出是否符合事实/任务要求 | "Agent 给出的数据是否正确？" |
| 完整性 | 是否覆盖了应有的步骤 | "Agent 是否遗漏了必要的验证环节？" |
| 合理性 | 推理路径是否自洽 | "Agent 的每一步推理是否在逻辑上成立？" |
| 工具使用质量 | 工具调用是否恰当（时机、参数） | "search 被调用时参数是否准确？" |
| 潜在风险 | 是否存在可能引发问题的行为 | "这次输出是否包含不安全的操作建议？" |

**⚠️ 已知风险（来自 critical-comparison.md）**：
- LLM 评委本身的偏好偏差和不一致性
- 递归问题：谁来验证验证者？
- 行业研究表明 LLM 评估者在不同模型间结果可能不一致
- **缓解建议**：引入模型多样性（Writer 用模型 A，Supervisor 用模型 B），对同一输出用多次打分取均值

#### 操作 2.3：评分标准定义模板

为每个 Happy Path 提供标准化的评分标准定义模板，确保每次评估一致。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **需人工** | 评分标准本身需要人来定义 |
| 实现复杂度 | **低** | 本质是一个结构化文档 + prompt 模板 |
| 技能要求 | **中** | 需要能把"感觉好"翻译成可量化的评分维度 |
| 独立价值 | **核心** | 没有好的评分标准，LLM 裁判就是乱打分 |
| 可实现性评分 | **8/10** | 文档化工作，无技术障碍 |

**模板示例**：
```yaml
test_case_id: "search_and_summarize_001"
expectation:
  trigger_condition: "用户提出需要搜索外部信息的查询"
  expected_path:
    - step: "理解查询意图"
    - step: "调用 search 工具"
      constraints: "query 参数应与用户意图匹配"
    - step: "整理搜索结果"
      constraints: "应引用至少 2 条搜索结果"
    - step: "生成带引用的总结"
  quality_criteria:
    accuracy: "引用的数据必须与搜索结果一致"
    completeness: "必须覆盖用户问题的所有子问题"
    risk: "不应编造搜索中不存在的数据"
scoring_rubric:
  1-3: "未触发搜索、编造数据、遗漏关键问题"
  4-6: "触发了搜索但引用不足、部分覆盖"
  7-9: "正确搜索和引用、完整覆盖、推理合理"
  10: "超出预期，包含额外洞察但未过度发散"
```

#### 操作 2.4：验证结果聚合器

将确定性断言的结果（Pass/Fail 列表）与 LLM 裁判的量化分数合并为一份综合报告。

| 可实现性评分 | **9/10** | 简单的数据拼接，无技术障碍 |

### 2.3 回归测试组件（4 个操作）

> 目标：积累可重复执行的测试用例库，每次改动自动复查

#### 操作 3.1：Happy Path 测试用例生成器

每加一个新功能时，根据功能描述自动生成 Happy Path 测试用例（触发条件 + 预期路径 + 预期结果）。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **半自动** | LLM 生成初稿 → 人工审核确认 |
| 实现复杂度 | **中** | 需要 prompt 模板 + 从功能描述到测试用例的生成 pipeline |
| 技能要求 | **中** | 开发者需要审核和修正 AI 生成的测试用例 |
| 独立价值 | **有用** | 降低编写测试用例的认知负担，但不替代人工判断 |
| 可实现性评分 | **6/10** | LLM 生成测试用例的质量取决于功能描述的清晰度 |

**生成模板（Happy Path）**：
```json
{
  "test_id": "auto_gen_20260630_001",
  "feature": "新增的搜索工具",
  "trigger": "用户查询中包含'搜索'、'查找'等关键词",
  "expected_workflow": [
    {"step": 1, "action": "解析搜索意图", "assert": "搜索 query 不为空"},
    {"step": 2, "action": "调用 search_tool", "assert": "tool_call 存在且参数正确"},
    {"step": 3, "action": "整合结果", "assert": "引用至少 2 条搜索结果"},
    {"step": 4, "action": "输出总结", "assert": "最终输出包含引用来源"}
  ],
  "assertions": {
    "hard": ["search_tool 必须被调用", "引用的数据条目数 ≥ 2"],
    "soft": ["总结的准确性", "推理的合理性"]
  }
}
```

#### 操作 3.2：真实输入收集器

从生产日志/用户交互中自动收集真实输入，清洗后加入测试套件。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **半自动** | 自动收集和清洗，但入库前建议人工抽查 |
| 实现复杂度 | **中** | 日志解析 + 去重 + 异常过滤 + PII 脱敏 |
| 技能要求 | **中** | 需要了解数据清洗和隐私合规 |
| 独立价值 | **核心** | 解决"开发者只能想到已知的测试场景"的盲区问题 |
| 可实现性评分 | **7/10** | 技术成熟，但隐私和数据质量是实际工程挑战 |

**收集策略**：
- 采集所有生产请求的输入
- 按频率采样（高频输入优先入库）
- 按异常采样（触发过错误的输入优先入库）
- 去重：对语义相似的输入聚类，每类保留代表
- 脱敏：移除 PII 数据（姓名、邮箱、电话等）

---

#### 操作 3.3：增量回归测试执行器

只对本次改动影响的测试用例子集运行回归测试（而非全量），缩短反馈时间。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 基于依赖分析自动选择测试范围 |
| 实现复杂度 | **高** | 需要建立代码变更 → 功能模块 → 测试用例的依赖图 |
| 技能要求 | **高** | 依赖分析在 AI Agent 这类不确定性系统上比传统软件更难 |
| 独立价值 | **有用** | 全量回归测试的 LLM 裁判成本很高时，增量运行是省钱的关键 |
| 可实现性评分 | **4/10** | AI Agent 的模块边界模糊，精确的依赖分析很难 |

**简化的增量策略**（实用版）：
- 如果只改了工具 X 的相关代码 → 只跑包含 tool_X 的测试用例
- 如果改了系统 prompt → 跑全量
- 如果改了 LLM 裁判的 prompt → 只跑 LLM 裁判，不跑断言

---

#### 操作 3.4：覆盖率缺口分析器

每次回归测试后，输出"测了什么"vs"还有什么没测"的差距报告。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 基于测试用例的元数据自动分析 |
| 实现复杂度 | **中** | 需要定义覆盖模型（功能覆盖点、路径覆盖点、输入类型覆盖） |
| 技能要求 | **高** | 定义"什么算覆盖"在 AI Agent 场景下需要深刻领域理解 |
| 独立价值 | **核心** | 解决"不知道测得够不够"的结构性焦虑 |
| 可实现性评分 | **5/10** | 覆盖模型的定义是关键难点--与芯片的功能覆盖率定义完全不同 |

**覆盖维度建议**：

| 覆盖维度 | 说明 | 衡量方式 |
|----------|------|----------|
| 功能覆盖 | 每个功能有没有 Happy Path 测试？ | 功能数 vs 有测试的功能数 |
| 工具覆盖 | 每个工具是否在各种输入参数组合下被测试？ | 工具×参数组合矩阵 |
| 路径覆盖 | workflow 的主要分支是否都被走过？ | 状态转移图中的边覆盖 |
| 输入类型覆盖 | 短查询/长查询/多轮/单轮/含错误输入 | 输入分类统计 |
| 异常覆盖 | 错误输入、超时、工具不可用场景 | 异常类型矩阵 |

---

### 2.4 Feature Flag 组件（2 个操作）

> 目标：每个新功能可独立开关，量化每次改动的影响

#### 操作 4.1：Feature Flag 配置管理器

为 Agent 的每个功能模块提供独立的开关标记，支持运行时动态切换。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 运行时配置读取，无需重启 |
| 实现复杂度 | **中** | 在 Agent 调度逻辑中插入 Flag 检查点 + Flag 存储（配置文件/环境变量/远程配置） |
| 技能要求 | **中** | 需要理解 Agent 的执行路径，在合适的位置埋 Flag 检查点 |
| 独立价值 | **核心** | 这是 A/B 对比测试的前提；也是灰度发布的基础设施 |
| 可实现性评分 | **8/10** | 标准 Feature Flag 实践，已有成熟方案 |

**Flag 粒度建议**：
- 工具级：`enable_new_search_tool`
- 行为级：`use_chain_of_thought_v2`
- 策略级：`max_search_results: 5 | 10 | 20`

---

#### 操作 4.2：A/B 对比报告生成器

对同一个测试套件分别以 Flag=ON 和 Flag=OFF 运行，生成差异对比报告。

| 维度 | 评估 | 说明 |
|------|------|------|
| 自动化程度 | **自动** | 两次运行 + diff 算法 + 报告模板 |
| 实现复杂度 | **低** | 本质是跑两次测试然后 diff |
| 技能要求 | **低** | 理解报告产出即可 |
| 独立价值 | **核心** | 这是回答"新功能到底带来了什么"的唯一方式--视频作者称之为"特别关键的实践" |
| 可实现性评分 | **9/10** | 技术上很简单：跑两次 + 对比 |

**报告格式（建议）**：

```markdown
## Feature Flag A/B 对比：new_search_tool

| 维度 | Flag=OFF (旧) | Flag=ON (新) | 变化 |
|------|--------------|-------------|------|
| 断言通过率 | 12/15 (80%) | 14/15 (93%) | **+13%** |
| LLM 评分均值 | 6.8/10 | 7.9/10 | **+1.1** |
| 平均执行步数 | 5.2 | 4.1 | **-1.1** (更高效) |
| 平均耗时 | 12.3s | 8.7s | **-3.6s** |
| 搜索工具调用率 | 0% | 87% | **+87%** (预期内) |

### 变好的项
- 数据准确率从 72% 提升到 91%
- 响应更简洁（减少了冗余的输出步骤）

### 变坏的项（Regression）
- `simple_query_003`：新工具在无搜索意图的查询中被误触发，LLM 评分下降 2.1
- `error_handling_007`：搜索超时时缺少降级处理

### 结论
新工具整体正向，但需要修复误触发和超时降级两个 regression
```

## 四、优先级排序的工程组件清单

排序原则：**(独立价值 × 可实现性) / 实现复杂度**，优先做"做了立刻有用、而且不复杂"的事。

### P0：立刻做（无任何依赖，立竿见影）

| 排序 | 组件 | 理由 |
|------|------|------|
| **1** | **1.2 结构化中间态日志** | 零依赖、低复杂度、高可实现性（9/10）。没有结构化日志，所有后续验证组件都没有数据源。**1-2 天搞完，立刻能用。** |
| **2** | **2.1 确定性断言引擎** | 基于结构化日志，立刻能写硬性检查规则。零成本（无 LLM API 消耗）、绝对可靠。**半天写引擎 + 半天写 3 条断言，当天见效。** |
| **3** | **2.3 评分标准定义模板** | 文档化工作，不需要写代码。先定义"什么算对"，后续 LLM 裁判才有依据。**半天写好模板。** |

### P1：尽快做（需少量依赖，但独立价值极高）

| 排序 | 组件 | 理由 |
|------|------|------|
| **4** | **1.1 Agent CLI 封装器** | 依赖对现有 Agent 入口的理解。做完后，回归测试就能挂接。复杂度中等但价值极高。**1-2 天。** |
| **5** | **4.1 Feature Flag 配置管理器** | 依赖 Agent 调度代码中能插入检查点。不做 FLAG，A/B 对比无从谈起。**1 天。** |
| **6** | **4.2 A/B 对比报告生成器** | 依赖 Feature Flag 有了 + 回归测试能跑了。技术简单（跑两次 + diff），但回答"新功能到底带来了什么"这个核心问题。**半天。** |

### P2：重要但需要投入（解决核心不确定性问题）

| 排序 | 组件 | 理由 |
|------|------|------|
| **7** | **2.2 LLM 裁判/评分器** | 复杂度高（5/10 可实现性）、API 成本、评委可靠性问题。但它是对 AI Agent 这种不确定系统进行模糊质量评估的**唯一途径**。没有它，断言只能覆盖 30-50% 的问题。**建议先用最简单的单轮打分，逐步迭代。** |
| **8** | **3.2 真实输入收集器** | 解决"开发者想象不到"的测试盲区。但需要生产流量才能发挥作用。**有流量就做，没流量先跳过。** |

### P3：锦上添花（有更好，但不是生死攸关）

| 排序 | 组件 | 理由 |
|------|------|------|
| **9** | **1.3 ACI 接口规范** | 对 Codex 闭环场景非常重要（SWE-Agent 论文证明接口 > 模型），但对人类开发者直接使用不是必须。**打算接 Codex 闭环时再做。** |

| **11** | **2.4 结果聚合器** | 断言结果和 LLM 打分分散输出也能看。聚合报告是体验优化，不影响功能。 |
| **12** | **1.4 健康检查** | 标准运维实践，但和"验证体系"的核心价值关联较弱。顺手做了也行。 |

## 五、按用户场景的工作流设计

### 场景 A：「我要给 Agent 加一个新功能，如何验证它没有退化？」

```
触发条件：新增工具/修改行为 → 需要验证
适用组件：1.1, 1.2, 2.1, 2.2, 2.3, 3.1, 4.1, 4.2

Step 1: 定义 Happy Path 验收标准 ─────────────────────────── 工具: 2.3
  ├── 填写评分标准模板：触发条件、预期路径、质量维度、评分 rubric
  └── 产出：一份 Happy Path 定义文档
       ⚡ 耗时：~15 分钟/功能

Step 2: 设置 Feature Flag ────────────────────────────────── 工具: 4.1
  ├── 为新功能添加一个 Flag（如 enable_new_xxx），默认 OFF
  └── 产出：Flag 配置项
       ⚡ 耗时：~5 分钟

Step 3: 生成 Happy Path 测试用例 ─────────────────────────── 工具: 3.1
  ├── 可选：用 LLM 从 Happy Path 定义自动生成测试用例 JSON
  ├── 人工审核并修正
  └── 产出：1 个新的测试用例（加入回归套件）
       ⚡ 耗时：~10 分钟

Step 4: 写确定性断言 ─────────────────────────────────────── 工具: 2.1
  ├── 工具触发断言（如 "search_tool 必须被调用"）
  ├── 参数检查断言（如 "search query 非空"）
  └── 产出：断言配置
       ⚡ 耗时：~10 分钟

Step 5: 实现功能代码 ───────────────────────────────────────
  └── 正常开发
       ⚡ 耗时：取决于功能复杂度

Step 6: 跑 A/B 对比测试 ──────────────────────────────────── 工具: 4.2, 1.1, 1.2
  ├── Flag=OFF：跑全量回归测试套件 → 基线分数
  ├── Flag=ON ：跑全量回归测试套件 → 新版本分数
  └── 产出：对比报告（哪些变好，哪些变坏）
       ⚡ 耗时：取决于测试套件大小和 LLM 裁判延迟

Step 7: 审查回归报告 ─────────────────────────────────────── 工具: 2.2, 2.4
  ├── 断言层：有没有新增 FAIL？
  ├── LLM 评分层：哪些测试用例分数下降了？
  └── 决策：
      ├── 全部通过且分数不降 → ✅ 完成
      ├── 有 FAIL → 🔧 修复代码 → 回到 Step 6
      └── 分数明显下降但无法立即修复 → ⚠️ 记录已知退化，评估是否阻挡发布

验证标准：
  ✅ 0 个新增断言失败
  ✅ LLM 评分在全量回归测试中下降 < 0.3（均值）
  ✅ 新功能 Happy Path 断言全部通过
  ✅ A/B 对比报告明确列出了"变好"和"变坏"的项
```

### 场景 B：「我修改了 prompt，如何知道是变好还是变坏？」

```
触发条件：修改了系统 prompt/工具描述/行为指引
适用组件：1.1, 1.2, 2.1, 2.2, 2.4

Step 1: 固定测试输入集 ───────────────────────────────────── 工具: 1.1, 1.2
  ├── 选取 5-10 个代表性测试用例（覆盖典型场景和边缘场景）
  └── 产出：固定的输入集
       ⚡ 耗时：~5 分钟（从已有回归套件中选取）

Step 2: 跑旧 prompt 基线 ─────────────────────────────────── 工具: 1.1, 2.1, 2.2
  ├── 对固定输入集跑一次完整测试
  ├── 记录：每个用例的断言结果 + LLM 评分 + 执行轨迹
  └── 产出：基线 JSON
       ⚡ 耗时：取决于用例数

Step 3: 修改 prompt ────────────────────────────────────────
  └── 正常修改
       ⚡ 耗时：取决于修改范围

Step 4: 跑新 prompt ──────────────────────────────────────── 工具: 1.1, 2.1, 2.2
  ├── 同样的输入集，跑一次完整测试
  └── 产出：新版本 JSON
       ⚡ 耗时：同 Step 2

Step 5: 逐条对比 ─────────────────────────────────────────── 工具: 2.4
  ┌────────────────────────────────────────────────────────┐
  │ 用例          旧断言  新断言  旧评分  新评分  变化     │
  │ simple_001     PASS    PASS    8.2     7.9    -0.3    │
  │ complex_005    PASS    PASS    6.1     8.7    +2.6 ✓  │
  │ edge_012       FAIL    PASS    3.2     7.1    +3.9 ✓  │
  │ typical_003    PASS    FAIL    7.5     4.1    -3.4 ⚠  │
  └────────────────────────────────────────────────────────┘

Step 6: 深入分析变坏的项 ─────────────────────────────────── 工具: 1.2
  ├── 对比两次执行轨迹（结构化日志）
  ├── 识别行为差异：哪里走了不同的路径？哪个工具调用变了？
  └── 产出：根因分析
       ⚡ 耗时：~10 分钟/条

Step 7: 决策
  ├── 全面变好或基本持平 → ✅ 接受修改
  ├── 有显著变坏的项 → 🔧 分析根因 → 调整 prompt → 回到 Step 4
  └── 好坏参半 → ⚖️ 权衡后决定（记录 tradeoff）

验证标准：
  ✅ 80% 以上用例评分不下降
  ✅ 0 个新增断言失败
  ✅ 任何评分下降 > 2.0 的用例必须有根因分析和明确的 tradeoff 记录
```

    ab-comparison/           # 4.2 A/B 对比报告生成器

  # P2 重要
  skills/
    llm-supervisor/          # 2.2 LLM 裁判/评分器
    input-collector/         # 3.2 真实输入收集器

  # P3 锦上添花
  skills/

### 6.2 最小可行 Skill（MVP）

如果不做全部，以下 **3 个 Skill** 单独实现就能产生 80% 的价值：

1. **`structured-log`**：注入结构化日志到 Agent 运行中
2. **`assertion-engine`**：定义并执行确定性断言
3. **`regression-run`**：串联 CLI 执行 + 断言检查 + 报告输出

这三个加起来，就能把"打开网页 → 手动输入 → 盯着看 → 凭感觉判断"变成"终端一条命令 → 自动输出 Pass/Fail"。

## 七、风险与假设

### 7.1 本分析基于的假设

| 假设 | 置信度 | 风险 |
|------|--------|------|
| 目标 Agent 有明确的输入→输出行为链路（不是完全开放的对话） | 高 | 纯创意/对话 Agent 不适用双层裁判 |
| 开发者可以接受 LLM 裁判的 API 成本 | 中 | 大规模回归测试的 Token 消耗可能很高 |
| Agent 的内核是可控代码而非纯第三方 API | 中 | 如果 Agent 是 SaaS 产品（非自建），很多改造无法进行 |
| 团队有 1 人以上投入验证体系建设 | 中 | 单人项目可能无法承受全量体系的维护成本 |

### 7.2 关键未解决问题（需要后续研究）

1. **LLM 裁判的信度和效度**：同一输入多次打分的一致性有多高？裁判的评分与真实用户满意度相关性如何？
2. **断言规则与 LLM 评分的仲裁机制**：断言 Pass 但 LLM 打分很低时怎么办？反之呢？
3. **覆盖率模型**：AI Agent 的"行为覆盖率"如何定义和衡量？传统代码覆盖率显然不够。
4. **模型升级的影响**：底层 LLM 升级后，已有测试用例需要多大比例的重新校准？
5. **闭环收敛的可靠性**：Codex 自我修复的成功率？是否会死循环？

## 八、总结

从视频的四个大组件（前后端分离、双层裁判、回归测试、Feature Flag）中，提取了 **16 个可工程化操作**。经过可实现性、独立价值、复杂度的三维修正，优先级排序如下：

**路线图速览**：
```
第 1 周（P0）：结构化日志 → 确定性断言引擎 → 评分标准模板
第 2 周（P1）：CLI 封装器 → Feature Flag 管理器 → A/B 对比报告
第 3-4 周（P2）：LLM 裁判（先简单版本）→ 真实输入收集器
第 2 个月+（P3）：ACI 接口 → Happy Path 生成器 → 覆盖率分析
```

**关键原则**：
- 每个组件都可以独立使用（不强依赖全套体系）
- 从 3 个 Happy Path 和 3 条断言开始，不要一开始就建大全套
- 作者的核心洞察不是工具本身，而是**先定义"什么算对"，再让自动化接手**
- 如果只有一个人，做 P0 + P1 就足以把 90% 手动测试中的大部分转化为自动化
